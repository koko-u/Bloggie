# AGENTS.md

このファイルは、このリポジトリで作業する AI エージェント向けのガイドです。

## 最重要ルール

- ユーザーから明示的にコード変更を依頼されていない限り、コードを変更しないこと。
- 調査、原因説明、レビュー、方針相談の依頼では、ファイル編集を行わず、必要なら変更案を提示するだけにすること。
- 編集が必要だと判断した場合でも、ユーザーの依頼範囲を超える修正、ついで修正、リファクタリングを行わないこと。
- 既存の未コミット変更はユーザーの作業として扱い、勝手に戻さないこと。
- 破壊的な操作や大きな生成物の削除は、必ず事前にユーザーの明示的な許可を得ること。

## プロジェクト概要

- ソリューションルート: このファイルがあるディレクトリ
- ソリューション: `Bloggie.slnx`
- Web プロジェクト: `Web/Bloggie.Web.csproj`
- フレームワーク: ASP.NET Core / .NET `net10.0`
- UI: Razor Pages
- DB アクセス: Dapper / Npgsql
- SQL ファイル: `Web/Sql/**/*.sql`
- 静的ファイル: `Web/wwwroot`

## よく使うコマンド

- ビルド: `dotnet build Web/Bloggie.Web.csproj`
- 実行: `just run`
- Watch 実行: `just watch`
- 整形: `just fmt`

`just` が使えない環境では、次を使う。

- 実行: `dotnet run --project Web --launch-profile https`

## 実装方針

- 既存のディレクトリ構成、命名、実装パターンに合わせること。
- ASP.NET Core の標準機能を優先し、不要な独自抽象化を追加しないこと。
- Razor Pages では PageModel、Form モデル、Row モデル、Service、SQL ファイルの責務を混ぜないこと。
- SQL は既存どおり `Web/Sql` 配下に置き、文字列連結で SQL を組み立てないこと。
- Dapper のパラメータ化を使い、ユーザー入力を直接 SQL に埋め込まないこと。
- `Nullable` が有効なので、null 許容性を明示し、警告を安易に抑制しないこと。
- `ImplicitUsings` は無効なので、必要な `using` は明示すること。

## ファイルとパス

- ファイルシステムのパスには `Path.Combine` を使うこと。
- URL 文字列とファイルシステムパスを混同しないこと。
- `wwwroot` 配下へ保存するファイルの実パスは `IWebHostEnvironment.WebRootPath` を基準にすること。
- ブラウザに返す URL は `/uploads/...` のような URL として別に組み立てること。

## 検証

- コードを変更した場合は、原則として `dotnet build Web/Bloggie.Web.csproj` を実行すること。
- 整形だけを行う場合は `just fmt` を使うこと。
- ビルドや検証を実行できなかった場合は、理由をユーザーに明示すること。

## コミュニケーション

- まず調査して、根拠となるファイルや行を示すこと。
- 変更前に、何を変更するかを短く説明すること。
- ユーザーが質問だけをしている場合は、回答に徹し、実装しないこと。
- 日本語で依頼された場合は、日本語で簡潔に回答すること。
